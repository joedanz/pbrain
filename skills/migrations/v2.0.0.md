# v2.0.0 Migration: Markdown-First Storage

This migration makes the filesystem authoritative. Your `.md` files are the
brain. The PGLite (or Postgres) database is a rebuildable index on top of
them. Nothing you own moves — this migration only affects where PBrain
looks for source-of-truth and where it writes.

**This migration does NOT touch existing content.** It updates config shape,
default index location for new installs, and write primitives. Existing
data and existing brains keep working as-is.

## What Changed

### Config: new `brain_path` field

`PBrainConfig` gains an optional `brain_path` — the absolute path to your
markdown brain folder. Any filesystem path works: local folder, Google
Drive Desktop mount, iCloud Drive folder, or an existing Obsidian vault.

Set it three ways:
- On `pbrain init`: interactive prompt, or `--brain-path <dir>` flag
- Env var: `PBRAIN_BRAIN_PATH` (for non-interactive scripting)
- Edit `~/.pbrain/config.json` directly

Existing installs do NOT have `brain_path` set. Commands that need a brain
folder still accept a positional arg (e.g., `pbrain index <dir>`). Setting
`brain_path` just means you can drop the positional arg.

### Default index location (new installs only)

New installs land the PGLite index at `~/.pbrain/indexes/default.pglite`
instead of `~/.pbrain/brain.pglite`. Reason: keeping the index OUT of the
brain folder matters because binary PGLite files corrupt under cloud sync.
Having `~/.pbrain/indexes/` also lets you keep multiple indexes pointing
at different brain folders later without naming collisions.

**Existing installs keep using `~/.pbrain/brain.pglite`.** No migration
needed — `pbrain init` detects the legacy file and preserves it.

### Atomic writes

All PBrain-written markdown files now go through `atomicWriteFileSync`:
write to a sibling `.pbrain-tmp-<uuid>` file, `fsync`, then rename into
place. Obsidian (or any reader) sees either the old contents or the new
contents — never a half-written file.

### 60-second modification cooldown

The new `writePageFile` helper in `src/core/page-writer.ts` is the single
chokepoint for writing brain pages. It checks the target file's mtime:
if the file was modified within the last 60 seconds, the user is probably
editing it in Obsidian right now. The write defers (returns
`{ status: 'deferred' }`); the next autopilot cycle picks it up after
the user stops typing. Callers can pass `{ force: true }` to override
when they're certain the write is safe.

### Wikilink and tag-footer primitives

Two new utilities for emitting Obsidian-native metadata:

- `src/core/wikilink.ts` — emit `[[slug]]`, parse, resolve against known
  slugs + `aliases:` frontmatter. Also exports `toPlainMarkdown` for
  downstream tools that don't resolve wikilinks.
- `src/core/tag-footer.ts` — write tags in both YAML frontmatter AND an
  inline `<!-- pbrain-tags -->\n#tag #tag` footer. Belt-and-suspenders
  so every consumer (Obsidian search, Dataview, PBrain's parser, GitHub
  render) finds them without parsing YAML.

### `pbrain index` command

`pbrain index` is the new name for `pbrain import`. Semantically clearer
now that files are authoritative — you're rebuilding the *index*, not
*importing* anything. `pbrain import` stays as an alias, so existing
scripts and docs keep working.

If `brain_path` is set in config, `pbrain index` with no positional arg
walks that folder by default.

## Agent Instructions

### Step 1: Pull the updated code

No-op if you're syncing normally. The changes are additive.

### Step 2: Optionally set `brain_path`

If you have a markdown brain folder you want PBrain to treat as
authoritative, set it once:

```bash
# Non-interactive
pbrain init --brain-path ~/my-brain --pglite

# Or for existing installs, edit the config directly
# ~/.pbrain/config.json
{
  "engine": "pglite",
  "database_path": "...",
  "brain_path": "/Users/you/my-brain"
}
```

### Step 3: Try `pbrain index`

```bash
# With brain_path set
pbrain index

# Or explicitly
pbrain index ~/my-brain
```

Behaves identically to `pbrain import` — same chunking, embedding, and
indexing pipeline. The rename is signaling, not semantic.

### Step 4: Log migration complete

```bash
mkdir -p ~/.pbrain/migrations
echo '{"version":"2.0.0","ts":"'$(date -u +%Y-%m-%dT%H:%M:%SZ)'","status":"complete"}' >> ~/.pbrain/migrations/completed.jsonl
```

## What DOESN'T change

- Your existing `.md` files: untouched
- Your existing PGLite or Postgres data: untouched
- `pbrain sync`, `pbrain query`, `pbrain enrich`: unchanged behavior
- Skills that write to the DB via `put_page`: unchanged behavior

Writing `.md` files directly to disk is a future move for skills. This
migration puts the primitives in place; skills adopt them over subsequent
versions on a case-by-case basis.

## Rollback

No content changes. To roll back:
1. Restore the code to pre-v2.0.0 state from git history.
2. Remove the completion marker:
   `grep -v '"version":"2.0.0"' ~/.pbrain/migrations/completed.jsonl > /tmp/m && mv /tmp/m ~/.pbrain/migrations/completed.jsonl`
3. If you set `brain_path` and want to remove it: delete the field from
   `~/.pbrain/config.json`. Nothing downstream requires it.

Safe by construction — this migration adds capabilities rather than
mutating existing ones.
