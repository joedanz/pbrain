# v0.3.0 Migration: Bi-temporal Links

One required step. Run it once. Takes under a second.

```bash
pbrain upgrade
pbrain apply-migrations --migration 0.3.0
```

## What changed

The `links` table gets two new columns: `valid_from DATE` and `valid_until DATE`.

Before: removing a link deleted the row. The history was gone.

After: removing a link sets `valid_until = today` and keeps the row. The graph
holds the closed period as history. All existing queries (`getLinks`, `getBacklinks`,
`traverseGraph`) filter to `valid_until IS NULL` — current edges only. Nothing
changes for callers who don't use the new columns.

The named UNIQUE constraint is replaced with a partial unique index that enforces
uniqueness only among current edges (`WHERE valid_until IS NULL`). This allows
multiple historical rows for the same (from, to, type) triplet while still
preventing duplicate *current* edges.

## What the migration does

Migration v11 (`bitemporal_links`) runs two steps:

1. **Schema**: Adds `valid_from DATE NULL` and `valid_until DATE NULL` to `links`.
   Existing rows get `NULL` for both (honest: unknown provenance, still current).
   Drops the old `UNIQUE(from_page_id, to_page_id, link_type)` constraint and
   creates `CREATE UNIQUE INDEX idx_links_current_unique ON links(from_page_id,
   to_page_id, link_type) WHERE valid_until IS NULL`.

2. **Verify**: Confirms both columns exist and the partial index is present.

Idempotent — safe to re-run. No data loss. Transactional.

## Verification

After migration completes:

```bash
# Confirm schema version is at v11
pbrain config get version
# Expect: 11 (or higher if later migrations have run)

# Confirm the brain is healthy
pbrain doctor
# No schema errors should appear
```

## New capabilities (optional)

Once the schema is applied, your agent can use two new parameters:

- **`add_link` — `valid_from` date**: Record when a relationship actually started.
  Pass `2019-06-01` and the brain knows Alice joined Google in June 2019, not just
  "whenever this link was inserted." Omit it and `valid_from` stays `NULL` (honest:
  unknown provenance).

- **`remove_link` — `link_type`**: Soft-close only one relationship type.
  Omit it to close all types between two entities; pass `works_at` to close only
  employment while leaving `invested_in` open.

No existing code needs to change. The new params are optional.
